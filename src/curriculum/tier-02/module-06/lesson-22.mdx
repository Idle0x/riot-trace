## The Concept Behind React Hooks

When you write `const [count, setCount] = useState(0)`, React is using a closure to remember `count` between renders. The `count` variable persists because the setter function and the state itself are held together in a closure. Understanding closures is understanding the mechanism behind every hook you use.

A closure is created whenever a function is defined inside another function. The inner function **retains access to the outer function's variables even after the outer function has finished executing**. The outer function's scope does not disappear — the inner function holds a reference to it.

---

## The Anatomy of a Closure

```javascript
function createCounter() {
  let count = 0;  // this variable lives in createCounter's scope

  return function increment() {
    count++;       // increment can still access count
    return count;  // even after createCounter() has finished
  };
}

const counter = createCounter();
// createCounter() has returned — normally count would be garbage collected
// But counter holds a reference to increment, which holds a reference to count
// So count stays alive as long as counter exists

counter();  // 1
counter();  // 2
counter();  // 3
```

Three things happening here:
1. `createCounter()` is called — a new scope is created with `count = 0`
2. `createCounter()` returns — but the returned function `increment` keeps a live reference to `count`
3. Each subsequent call to `counter()` modifies the same `count` through the closure

---

## Each Call Creates a New Closure

Every time the outer function runs, a completely fresh scope is created. Multiple closures from the same factory are independent:

```javascript
const counter1 = createCounter();
const counter2 = createCounter();

counter1();  // 1
counter1();  // 2
counter2();  // 1 — independent! counter2 has its own count
counter1();  // 3 — counter1's count is still 3
```

This is what enables the factory and module patterns from Lesson 21. `createFilter(80)` and `createFilter(50)` each have their own closure over their own `threshold` value.

## The var-in-Loop Closure Trap

One of the most famous JavaScript bugs, still encountered in legacy code:

```javascript
// The trap:
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// After 100ms, logs: 3, 3, 3 (not 0, 1, 2)
```

Why? `var` is function-scoped — there is only ONE `i` variable shared by all three closures. By the time the timeouts fire (100ms), the loop has finished and `i` is `3`. All three functions close over the same `i`.

```javascript
// The fix — let creates a new binding per iteration:
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// After 100ms, logs: 0, 1, 2
```

With `let`, each loop iteration creates a new block scope with its own `i`. Each closure captures a different variable.

## The Stale Closure — React's Most Confusing Bug

Closures capture the variable binding, not a snapshot of the value at a point in time. But this interacts subtly with reassignment:

```javascript
let name = "riot";
const greet = () => `Hello, ${name}`;

name = "trace";
greet();  // "Hello, trace" — closure sees the current value of name
```

Arrow functions close over the variable binding itself. When `name` is reassigned, the binding still points to the updated value. `greet` sees `"trace"` because it reads `name` at call time, not at definition time.

**The React stale closure is different.** In React, state values are captured at render time. A `useEffect` callback closes over the value of state from the render when the effect was created:

```javascript
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const interval = setInterval(() => {
      console.log(count);    // STALE — always 0 from initial render
      setCount(count + 1);   // STALE — always sets to 0 + 1 = 1
    }, 1000);
    return () => clearInterval(interval);
  }, []);  // empty deps — effect runs once, closes over count = 0
}
```

The `count` inside the interval callback is not the live `count` variable — it is the value of `count` when the effect ran (0). Even as state updates, the closure sees the original snapshot.

**Fix 1 — Updater function (most common):**
```javascript
setCount(prev => prev + 1);  // prev is always current — doesn't use closure
```

**Fix 2 — Add count to dependencies (re-creates the closure on change):**
```javascript
useEffect(() => {
  const interval = setInterval(() => {
    setCount(count + 1);  // now current for each render
  }, 1000);
  return () => clearInterval(interval);
}, [count]);  // new closure when count changes
```

**Fix 3 — useRef (persists without triggering re-renders):**
```javascript
const countRef = useRef(count);
countRef.current = count;  // always current — updated on every render
useEffect(() => {
  const interval = setInterval(() => {
    setCount(countRef.current + 1);  // reads from ref, always fresh
  }, 1000);
}, []);
```

You will use all three patterns in Tier 3. Recognizing which pattern fits which situation comes from understanding the closure mechanics here.

## Private State via Closures — The Module Pattern

Closures are the original mechanism for private state in JavaScript (before `class` and `#private` fields). The outer function's variables are completely inaccessible from outside — they are private by nature:

```javascript
function createNodeMonitor(nodeId) {
  let healthHistory = [];  // private — nothing outside can access this directly
  let alertThreshold = 50; // private

  return {
    recordHealth(score) {
      healthHistory.push(score);
      if (score < alertThreshold) {
        return `ALERT: ${nodeId} health dropped to ${score}%`;
      }
      return `OK: ${nodeId} at ${score}%`;
    },
    setThreshold(value) {
      alertThreshold = value;
    },
    getHistory() {
      return [...healthHistory];  // return a copy — not the private array
    }
  };
}

const monitor = createNodeMonitor("7A5r");
monitor.recordHealth(95);       // "OK: 7A5r at 95%"
monitor.recordHealth(42);       // "ALERT: 7A5r health dropped to 42%"
monitor.healthHistory;          // undefined — private
monitor.getHistory();           // [95, 42] — copy via the public method
```

This pattern — private data, public interface — is the foundation of React's component state model. The `useState` hook hides the actual state variable and exposes only `count` and `setCount`.

## Common Mistakes

**Mistake 1 — The var-in-loop trap:**
```javascript
for (var i = 0; i < n; i++) {
  // closures here all share the same i
}
// Fix: use let
```

**Mistake 2 — Reinitializing state inside the returned function:**
```javascript
function makeCounter() {
  return function() {
    let count = 0;  // resets on every call — not a closure over shared state
    return ++count; // always returns 1
  };
}
// Fix: move let count = 0 to the outer function
```

**Mistake 3 — Expecting closures to capture values, not bindings:**
```javascript
let x = 5;
const getX = () => x;
x = 10;
getX();  // 10 — captures the BINDING (variable), not the value 5
// This is usually correct behavior — but surprising when not expected
```

**Mistake 4 — Sharing state unintentionally:**
```javascript
// If counter1 and counter2 share the same let count in an outer scope,
// incrementing one affects the other. Each closure must have its own
// variable from a fresh outer function call.
```
