## The Test

Module 06 covered the complete function toolkit: declarations and expressions, arrow syntax, higher-order functions that accept and return functions, and closures that give functions persistent private memory.

This boss fight uses all five areas in a single coherent system. Each requirement demonstrates a distinct pattern — and they are designed so that confusing any two of them will produce wrong output.

---

## What This Tests

**Requirement 1** tests pure function discipline. `scoreNode` receives all its inputs as parameters, performs a calculation, and returns a result. No external state accessed, no mutations, no side effects. `.toFixed(1)` returns a string — the validator checks for the string `"82.5"`, not the number `82.5`.

**Requirement 2** tests the factory pattern. `createClassifier` is called once with a threshold, and returns a function. That returned function is then called with a node. The two steps happen separately — build the classifier, then apply it. Writing the classification inline instead of returning a function will fail the constraint check.

**Requirement 3** tests passing functions as arguments. `countWhere` is a general-purpose function — it has no knowledge of nodes, health, or any domain concept. It takes any array and any predicate. You pass the domain knowledge as a callback argument.

**Requirement 4** tests the closure for private state pattern. The XP bank's `balance` variable must be in the closure scope — not a property on the returned object. After three operations (deposit 300, deposit 250, withdraw 50), the balance is 500.

**Requirement 5** is the critical test of closure independence. `createModule` must be called twice, producing two objects backed by completely separate private variables. Any shared state between them — caused by declaring the tracking variables outside `createModule` — will cause Module-02 to show Module-01's data. The proof is simple: Module-02's report must show 0 lessons and 0 XP even though Module-01 has 2 lessons and 140 XP.

---

## Approach

Build and verify each requirement in sequence. Requirements 4 and 5 use similar patterns — build 4 first, then use the same closure technique for 5.

The most likely failure point: Requirement 5 state independence. If you declare `lessonsComplete` and `totalXP` in an outer scope and reuse them for both module instances, both modules will share those variables. Each call to `createModule` must create its own fresh variables inside its own execution context.
