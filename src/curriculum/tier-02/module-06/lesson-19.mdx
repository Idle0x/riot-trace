## The Unit of Reuse

Every concept you have learned so far — variables, conditionals, array methods — operates on single values or single collections. Functions are how you package that logic so it can be named, reused, tested, and composed.

This lesson is not your first encounter with functions — you have been writing arrow function callbacks since Lesson 15. But this lesson formalizes the complete picture: the two ways to declare a function, how parameters and return values work precisely, how default parameters prevent `undefined` bugs, and the habits that separate readable functions from fragile ones.

---

## Two Ways to Declare a Function

### Function Declaration

```javascript
function greet(name) {
  return `Hello, ${name}`;
}
```

Characteristics:
- Starts with the `function` keyword
- Has a name immediately after `function`
- Is **hoisted** — can be called before it appears in the source code
- Belongs to the enclosing scope (function or global)

### Function Expression

```javascript
const greet = function(name) {
  return `Hello, ${name}`;
};
```

Characteristics:
- Assigned to a variable (usually `const`)
- Is **not hoisted** — calling before the declaration throws a `ReferenceError`
- The function itself can optionally have a name (a named function expression), useful for recursion and debugging

### Arrow Function (previewed here, covered fully in Lesson 20)

```javascript
const greet = (name) => `Hello, ${name}`;
```

Modern syntax for function expressions. You have been writing these since Lesson 15. Lesson 20 covers every subtlety.

## Parameters — Complete Anatomy

Parameters are the named placeholders in a function definition. Arguments are the values passed when calling the function.

```javascript
// Definition: health and network are PARAMETERS
function describeNode(health, network) {
  return `${network} node at ${health}%`;
}

// Call: 95 and "mainnet" are ARGUMENTS
describeNode(95, "mainnet");
// "mainnet node at 95%"
```

### What Happens When You Pass the Wrong Number

JavaScript does not throw an error for parameter count mismatches:

```javascript
function add(a, b) {
  return a + b;
}

add(5, 3)        // 8   — correct
add(5)           // NaN — b is undefined, 5 + undefined = NaN
add(5, 3, 99)    // 8   — extra argument 99 is silently ignored
```

Missing parameters become `undefined`. This is why default parameters exist.

### Default Parameters

```javascript
function classifyNetwork(network = "mainnet") {
  return network === "mainnet" ? "production" : "non-production";
}

classifyNetwork("devnet")   // "non-production" — argument provided
classifyNetwork()           // "production" — no argument, default used
classifyNetwork(undefined)  // "production" — undefined triggers default
classifyNetwork(null)       // "non-production" — null does NOT trigger default
```

Default parameters only activate when the argument is `undefined`. Explicit `null` does not trigger them. This matches the distinction from Lesson 4.

## Return Values — The Contract

A function's return value is what it produces for the caller. It is the function's output.

```javascript
function getStatus(health) {
  if (health >= 80) return "HEALTHY";
  if (health >= 50) return "DEGRADED";
  return "CRITICAL";
}

const status = getStatus(95);   // "HEALTHY"
console.log(getStatus(45));     // logs "CRITICAL"
```

**Key rules:**

1. **`return` immediately exits the function.** No code after `return` runs in that call:

```javascript
function check(health) {
  return health > 80;
  console.log("this never runs");  // unreachable
}
```

2. **A function without `return` returns `undefined`:**

```javascript
function noReturn(x) {
  x * 2;  // not returned
}
noReturn(5);  // undefined
```

3. **Multiple `return` statements are valid** — often used for early exits:

```javascript
function getNodeInfo(node) {
  if (!node) return null;         // guard clause — early exit
  if (!node.id) return "no-id";  // another guard
  return `Node: ${node.id}`;     // normal path
}
```

The guard clause pattern (checking for invalid input first and returning early) keeps the "happy path" code clean and avoids deeply nested `if` blocks.

## Pure Functions vs Functions With Side Effects

A **pure function** returns a value based only on its inputs and does nothing else. Same inputs always produce the same output. No external state is modified:

```javascript
// Pure — predictable, testable, no side effects:
function calculateHealth(uptime, storage) {
  return (uptime * 0.35) + (storage * 0.30);
}
```

A **function with side effects** modifies something outside itself — logs to console, modifies external variables, makes network requests, reads from external state:

```javascript
// Side effect — modifies external variable:
let totalNodes = 0;
function registerNode(node) {
  totalNodes++;  // side effect
  console.log(`Registered: ${node.id}`);  // side effect
}
```

Both types are necessary and legitimate. The professional habit: **prefer pure functions wherever possible**. Pure functions are easier to test (call them with known inputs and check the output), easier to debug (the bug is always inside the function, not somewhere outside), and easier to understand at a glance.

## Function Scope and the return Statement

Each function call creates its own execution context — its own isolated scope. Variables declared inside a function are destroyed when the function returns:

```javascript
function processNode(node) {
  const tempScore = node.health * 0.7;    // exists only during this call
  const tempBadge = tempScore > 60 ? "A" : "B";  // same
  return { score: tempScore, badge: tempBadge };  // sent out via return
}

const result = processNode({ health: 95 });
// result: { score: 66.5, badge: "A" }
// tempScore and tempBadge are gone
```

The only way to get data out of a function is `return`. Everything else stays inside.

## The arguments Object — Legacy Awareness

Inside a regular function (not arrow), there is an implicit `arguments` object containing all arguments passed:

```javascript
function sum() {
  let total = 0;
  for (const arg of arguments) {
    total += arg;
  }
  return total;
}

sum(1, 2, 3, 4, 5);  // 15
```

This is legacy. Modern code uses rest parameters instead:

```javascript
function sum(...numbers) {
  return numbers.reduce((total, n) => total + n, 0);
}
sum(1, 2, 3, 4, 5);  // 15
```

`arguments` does not exist in arrow functions — one of several reasons arrow functions behave differently. You will see `arguments` in older codebases. Do not use it in new code.

## Common Mistakes

**Mistake 1 — Forgetting return:**
```javascript
function double(n) {
  n * 2;  // not returned — function returns undefined
}
const result = double(5);
console.log(result);  // undefined
```

**Mistake 2 — Return before logic is complete:**
```javascript
function calculate(a, b) {
  const sum = a + b;
  return sum;           // exits here
  const product = a * b;  // unreachable — never runs
  return product;
}
```

**Mistake 3 — Modifying parameters (objects are references):**
```javascript
function addBadge(node) {
  node.badge = "ELITE";  // modifies the ORIGINAL object
  return node;
}
// Fix — return a new object:
function addBadge(node) {
  return { ...node, badge: "ELITE" };
}
```

**Mistake 4 — Using null to trigger a default parameter:**
```javascript
function connect(url = "https://api.default.com") {
  console.log(url);
}
connect(null);      // "null" — null doesn't trigger default
connect(undefined); // "https://api.default.com" — undefined does
connect();          // "https://api.default.com" — same as undefined
```
