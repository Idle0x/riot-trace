JavaScript is **single-threaded**. It has only one lane of traffic. It can only do one thing at a time. So, what happens if it asks a database for 10,000 records? Does the entire app freeze while it waits? 

No. Because of the **Event Loop**.



### The Call Stack & The Queue
When JavaScript sees a command that takes time (like a network request or a timer), it hands that task off to the browser (or Node.js) and immediately moves on to the next line of code.

When the time-consuming task finishes, its result is placed in a **Queue**. The Event Loop constantly checks: *"Is the main lane empty? Yes? Okay, bring in the next item from the queue."*

### The 0ms Illusion
Look at this code:
```javascript
console.log("Start");

setTimeout(() => {
  console.log("Timer Finished");
}, 0); // 0 millisecond delay!

console.log("End");
```

Even though the delay is `0` milliseconds, the output will always be:
1. `Start`
2. `End`
3. `Timer Finished`

Why? Because `setTimeout` forces its code into the Queue. JavaScript *must* finish reading the entire main file (`Start`, then `End`) before it is allowed to look at the Queue.
