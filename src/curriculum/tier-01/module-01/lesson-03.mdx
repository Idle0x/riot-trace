## Why This Lesson Matters More Than It Appears

The most common error message in JavaScript is:

```
TypeError: Cannot read properties of null (reading 'health')
TypeError: Cannot read properties of undefined (reading 'id')
```

If you have used JavaScript for more than a week, you have seen this crash. It means you tried to access a property (`.health`, `.id`, `.name`) on a value that does not exist. That value is either `null` or `undefined`.

You have seen this in xandeum-pulse. The API call fires, the component renders before the data arrives, and the code tries to read `node.health` before `node` exists. The page crashes or goes blank. Understanding `null` and `undefined` precisely is understanding how to prevent, catch, and reason about this entire category of runtime error.

---

## undefined — The Automatic Absence

`undefined` is what JavaScript gives to a variable that has been declared but never assigned a value. You did not set it to `undefined`. JavaScript did. Automatically.

```javascript
let nodeStatus;
console.log(nodeStatus);   // undefined — no assignment was ever made
```

`undefined` means: **this variable exists, but nothing has been put into it.**

```javascript
// Anywhere JavaScript needs a value and has nothing to give you, it returns undefined:

let x;
console.log(x);           // undefined — declared, not assigned

function greet(name) {
  console.log(name);      // undefined — function called without that argument
}
greet();                  // no argument passed — name is undefined

const obj = { health: 95 };
console.log(obj.version); // undefined — property doesn't exist on the object

// typeof undefined:
typeof undefined          // "undefined"
```

The important thing: `undefined` is not an error. It is a value. It is the specific value JavaScript uses to represent "nothing was given here." The error comes when you try to do something with `undefined` that requires a different type — like accessing a property on it.

## null — The Intentional Absence

`null` is a value that **you assign** when you want to explicitly say "this intentionally has no value right now."

```javascript
let selectedNode = null;
// A developer wrote this.
// It means: "I know about selectedNode. It exists as a concept.
//            Right now, nothing is selected."

let currentUser = null;
// "The current user slot exists. Nobody is logged in yet."
```

The difference from `undefined`:

```javascript
let selectedNode;           // undefined — someone forgot to initialize this? Maybe.
let selectedNode = null;    // null — someone intentionally said "nothing selected yet."
```

`undefined` is what you get when JavaScript fills a gap. `null` is what you put when you are making a deliberate statement about the absence of a value.

In practice, you will use `null` at the beginning of component state when a value will eventually exist but does not yet:

```javascript
// In React (you will understand this fully in Tier 3):
const [selectedNode, setSelectedNode] = useState(null);
// "No node is selected yet. When the user clicks one, this will update."
```

## typeof null — The Famous JavaScript Bug

```javascript
typeof null     // "object"
```

This is wrong. `null` is not an object. It is a primitive value representing absence. But this has been in JavaScript since 1995, and changing it would break every website that relies on the (wrong) behavior. So it stays.

The practical consequence: **you cannot use `typeof` to check for null.**

```javascript
// WRONG — this check does not catch null
if (typeof selectedNode === "object") {
  console.log("it's null or an object — can't tell");
}

// CORRECT — use === for null checks
if (selectedNode === null) {
  console.log("it is definitively null");
}
```

There is one shortcut. Because `==` (loose equality) treats `null` and `undefined` as equal, you can check for both at once:

```javascript
if (selectedNode == null) {
  // This catches BOTH null AND undefined
  console.log("either null or undefined — not yet assigned");
}
```

This is one of the only legitimate uses of `==` instead of `===`. Use it when you genuinely do not care whether the value is `null` or `undefined` — only that it is one of those two.

## NaN — Revisited in Context

You were introduced to `NaN` in Lesson 2. Now it belongs in the same conversation as `null` and `undefined` because all three represent a kind of "emptiness" or "invalidity."

```javascript
0 / 0             // NaN
parseInt("error") // NaN
Math.sqrt(-9)     // NaN
undefined + 1     // NaN — arithmetic with undefined produces NaN
null + 1          // 1   — arithmetic with null coerces null to 0 (confusing!)
```

`NaN` is unique: it is the only value in JavaScript that is **not equal to itself**:

```javascript
NaN === NaN   // false
NaN == NaN    // false
```

The correct way to test for it:

```javascript
isNaN(value)           // true if value is NaN or can be coerced to NaN ("hello" → true)
Number.isNaN(value)    // true ONLY if value is exactly NaN (stricter, preferred)

Number.isNaN(NaN)      // true
Number.isNaN("hello")  // false — "hello" is a string, not NaN
Number.isNaN(0/0)      // true
```

## The Six Falsy Values — Complete Picture

Lesson 3 introduced truthy and falsy values. Now that you know `null`, `undefined`, and `NaN`, you can see the complete list of falsy values:

```javascript
false
0
-0
""          // empty string
null        // intentional absence
undefined   // uninitialized
NaN         // invalid number
```

Everything else is truthy. This list is exact and complete. There are no other falsy values in JavaScript.

Why does this matter? Because these values are what you check for in conditions:

```javascript
// All of these if blocks do NOT execute:
if (false) { }
if (0) { }
if ("") { }
if (null) { }
if (undefined) { }
if (NaN) { }

// This is why the guard pattern works:
if (selectedNode) {
  // This only runs if selectedNode is not null, not undefined, not 0, not ""
  console.log(selectedNode.health);  // safe — selectedNode exists
}
```

## The Crash and How to Prevent It

The error from the introduction of this lesson:

```
TypeError: Cannot read properties of null (reading 'health')
```

This happens when you write:

```javascript
let node = null;
console.log(node.health);
// JavaScript tries to access .health on null
// null has no properties
// Runtime error, page crashes
```

There are three standard patterns for preventing this:

### Pattern 1 — Guard Clause (if check)
```javascript
if (node !== null && node !== undefined) {
  console.log(node.health);
}

// Shorter version using truthiness:
if (node) {
  console.log(node.health);
}
```

### Pattern 2 — Optional Chaining (modern JavaScript)
```javascript
console.log(node?.health);
// The ?. is called optional chaining
// If node is null or undefined, this returns undefined instead of crashing
// If node exists, it accesses .health normally
```

```javascript
// Chaining deeper:
console.log(node?.location?.city);
// Returns undefined if node is null OR if node.location is null
// No crash either way
```

### Pattern 3 — Nullish Coalescing
```javascript
const health = node?.health ?? 0;
// ?? is the nullish coalescing operator
// If the left side is null or undefined, use the right side as the default
// If node?.health is undefined (node is null), health becomes 0
```

Optional chaining (`?.`) and nullish coalescing (`??`) are the two operators you will use the most when working with API data in React. Data from APIs is always potentially null or undefined — the data has not loaded yet, the field is optional, the record does not exist. These operators let you access nested values safely.

## Understanding the Bug in Your Own Code

Look at what happens in React when you fetch data:

```javascript
// Component renders BEFORE data arrives:
const [nodeData, setNodeData] = useState(null);  // starts as null

// Render runs with nodeData = null:
return <div>{nodeData.health}</div>;
// CRASH: Cannot read properties of null (reading 'health')
```

The fix:
```javascript
return <div>{nodeData?.health ?? 'Loading...'}</div>;
// If nodeData is null: shows 'Loading...'
// If nodeData exists: shows the health value
```

You have written code that handles this. Now you understand the exact mechanism behind why it was needed.

## typeof — The Complete Cheat Sheet

Now that you know all four primitive types from this module, here is `typeof` for all of them:

```javascript
typeof "hello"     // "string"
typeof 42          // "number"
typeof true        // "boolean"
typeof undefined   // "undefined"
typeof null        // "object"    ← the bug. null is not an object
typeof NaN         // "number"    ← NaN is of type number. Yes.
typeof []          // "object"    ← arrays are objects
typeof {}          // "object"    ← plain objects
typeof function(){} // "function" ← functions have their own typeof result
```

Notice: `typeof` is reliable for `string`, `number`, `boolean`, and `undefined`. For `null` it lies. For arrays and plain objects it gives the same result (`"object"`) and cannot distinguish them. You will need other checks (`Array.isArray()`, `=== null`) for those cases.

## Common Mistakes

**Mistake 1 — Treating undefined as an error:**
```javascript
let health;
console.log(health);  // undefined — this is valid, not a crash
// The crash comes from USING undefined, not from it existing
health + 10  // NaN — arithmetic with undefined
```

**Mistake 2 — Using typeof to check for null:**
```javascript
typeof null === "null"   // false — typeof null is "object", not "null"
typeof null === "object" // true — but this matches real objects too, unhelpful
null === null            // true — use strict equality for null checks
```

**Mistake 3 — Forgetting optional chaining when data is loading:**
```javascript
// This crashes if data is null:
return <div>{user.name}</div>

// This is safe:
return <div>{user?.name ?? 'Loading...'}</div>
```

**Mistake 4 — Confusing null coalescing ?? with OR ||:**
```javascript
const health = nodeHealth ?? 0;
// ?? only triggers for null and undefined
// If nodeHealth is 0 (a valid health score!), it stays 0

const health = nodeHealth || 0;
// || triggers for ANY falsy value
// If nodeHealth is 0, it becomes 0... wait, same result
// But if nodeHealth is "" (empty string), || replaces it, ?? does not
// ?? is stricter and safer for default values
```
