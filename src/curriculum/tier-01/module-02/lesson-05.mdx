## Why This Lesson Exists

Every piece of data your code works with needs a name. The node's health score, the active network, the XP total, the list of transactions — without names, you cannot reference values after you create them. Variables are those names.

But "variable" is a misleading word for something you use almost constantly. In modern JavaScript, most values do not vary. The node ID does not change. The API endpoint does not change. The tier name does not change. You declare them once and use them throughout. Understanding which keyword to use — and more importantly, *why* — is the difference between writing intentional code and writing code that happens to work.

This lesson also corrects a mental model that causes real bugs: the idea that `const` prevents change. It does not. It prevents something more specific. Understanding exactly what it prevents is the point of Task 2.

## The Three Declaration Keywords

JavaScript has three ways to declare a variable. Only two of them belong in your code.

### const

```javascript
const TIER_COUNT = 7;
const NODE_ID = "7A5rRdbGp4jUm4TA";
const network = "mainnet";
```

`const` declares a **binding** — a name that points to a value. The binding is permanent. You cannot point that name at a different value later:

```javascript
const health = 95;
health = 87;   // TypeError: Assignment to constant variable.
               // The name 'health' cannot be pointed at a different value.
```

Use `const` for everything that should not be reassigned. In practice, that is the vast majority of your variables. When you read code and see `const`, you know immediately: this name will always refer to the same value. That certainty makes code easier to reason about.

### let

```javascript
let score = 0;
let currentPage = 1;
let isLoading = true;
```

`let` declares a binding that *can* be reassigned. Use it specifically when the value will change:

```javascript
let xp = 0;
xp = xp + 300;    // Task 3 complete
xp = xp + 1500;   // Boss fight complete
console.log(xp);  // 1800
```

The restriction: `let` is **block-scoped**. It only exists inside the `{ }` block where it was declared. You will understand exactly what that means in the next lesson on scope.

### var — Why You Won't Use It

`var` is the original variable declaration from 1995. It exists in millions of codebases written before 2015, so you need to recognize it. You will not write it.

```javascript
var oldStyle = "this is from 1995";
```

`var` has three behaviors that cause bugs:

1. It is **function-scoped** instead of block-scoped — it leaks out of `if` blocks and `for` loops in ways that surprise you
2. It is **hoisted** in a way that silently gives variables the value `undefined` before they are initialized (covered in Lesson 7)
3. It allows **redeclaration** — you can accidentally declare the same variable twice with no error

Every place you would use `var`, use `const` or `let` instead. Your existing codebase uses `const` and `let` exclusively. Keep it that way.

## The Mental Model That Changes Everything

Here is the most important concept in this lesson, and the one that most tutorials skip:

**Variables are labels. Not boxes.**

The "box" mental model says: a variable is a container, and the value lives inside the container. This model breaks immediately when you work with objects and arrays.

The correct mental model: a variable is a **label attached to a location in memory**. The label points at the location. The value lives at that location.

```javascript
const a = [1, 2, 3];
//    ^         ^
//    label     location in memory where [1,2,3] is stored

const b = a;
//    ^   ^
//    another label pointing at THE SAME memory location
```

When you write `const b = a`, you do not copy the array. You attach a second label to the same memory location. Now both `a` and `b` point at the same array:

```javascript
const a = [1, 2, 3];
const b = a;
b.push(4);
console.log(a);  // [1, 2, 3, 4] — a changed!
// Because a and b are both labels for the same location.
// Pushing to b modified the array at that location.
// a still points at that same location, so it sees the change.
```

This is called **reference behavior**, and it applies to all objects and arrays. It does not apply to primitive values (strings, numbers, booleans), which are copied directly:

```javascript
let x = 5;
let y = x;   // y gets a COPY of the value 5, not a reference to x
y = 10;
console.log(x);  // 5 — x is unchanged, y got its own copy
```

## const Does Not Mean "Immutable"

This is the most common misconception about `const`, and it is the exact thing Task 2 tests.

`const` protects the **label** — it prevents the label from being pointed at a different location. It does not protect the **value** at that location from being changed.

```javascript
const node = { id: "abc", health: 95 };

// This FAILS — you are trying to point the label 'node' at a different location:
node = { id: "abc", health: 87 };
// TypeError: Assignment to constant variable

// This WORKS — you are modifying the object AT the current location:
node.health = 87;
console.log(node.health);  // 87
// The label 'node' still points at the same location.
// The content at that location has changed.
```

The distinction matters everywhere in React. You will write hundreds of `const` declarations for objects and arrays and then modify their properties without reassigning the variable itself. This is the correct pattern.

```javascript
// In xandeum-pulse, you might have:
const nodeStats = { mainnet: 147, devnet: 54, total: 201 };
nodeStats.total = 203;  // Valid — modifying the property
nodeStats.mainnet = 149; // Valid — modifying another property
// nodeStats = {};      // Invalid — would throw TypeError
```

## Initialization and Declaration

A declaration creates the variable. Initialization gives it its first value:

```javascript
// Declaration and initialization together (most common):
const health = 95;

// Declaration without initialization (only with let):
let status;
// status is now undefined — JavaScript's automatic "nothing here yet"
status = "online";  // initialization happens later
```

You cannot declare a `const` without initializing it. This is enforced:

```javascript
const health;  // SyntaxError: Missing initializer in const declaration
```

This makes sense. If `const` prevents reassignment and you never gave it a value, it would be permanently `undefined` with no way to change it. JavaScript prevents this at the syntax level.

## Naming Conventions — Professional Habits

The name you give a variable communicates intent. Professional codebases follow consistent conventions:

```javascript
// camelCase — for regular variables and functions (the standard in JavaScript)
const nodeHealth = 95;
const activeNodeCount = 201;
let isLoading = false;
let currentPage = 1;

// SCREAMING_SNAKE_CASE — for constants that are configuration values or never change
const MAX_RETRY_ATTEMPTS = 3;
const API_ENDPOINT = "https://api.xandeum.network";
const DEFAULT_TIMEOUT_MS = 15000;

// PascalCase — for classes and React components (you will use this in Tier 3)
const NodeCard = ...;
class NetworkOrchestrator { ... }
```

The distinction between `const nodeHealth` and `const MAX_RETRY_ATTEMPTS` is about communication. Both use `const`. But `MAX_RETRY_ATTEMPTS` in screaming snake case signals to every developer reading the code: this is a configuration constant. Do not touch it casually. `nodeHealth` is a regular variable that happens to be const.

Your existing code uses this convention. `NETWORK_ID`, `API_ENDPOINT`, `SUPABASE_URL` — all screaming snake case. `nodeData`, `currentTier`, `isCompleted` — all camelCase. Follow this pattern consistently.

## Common Mistakes

**Mistake 1 — Reassigning a const:**
```javascript
const health = 95;
health = 87;  // TypeError — use let if this value needs to change
```

**Mistake 2 — Thinking const prevents object mutation:**
```javascript
const config = { network: "mainnet" };
config = { network: "devnet" };   // TypeError — cannot reassign the label
config.network = "devnet";        // Valid — modifying the property at the location
```

**Mistake 3 — Using let for everything:**
```javascript
// Wrong — using let when the value never changes
let nodeId = "7A5rRdb";  // this ID never changes in this scope
let endpoint = "/api/stats";  // this never changes

// Correct — const communicates immutability of binding
const nodeId = "7A5rRdb";
const endpoint = "/api/stats";
```

When you use `const` for something that never changes, you are communicating to every reader of your code: "I verified this does not need to be mutable." That communication has value.

**Mistake 4 — Forgetting that var leaks out of blocks:**
```javascript
if (true) {
  var leaked = "I escaped the if block";
  let confined = "I stay in the if block";
}
console.log(leaked);   // "I escaped the if block" — var leaked out
console.log(confined); // ReferenceError — let stays inside the block
```

This is the block scoping behavior that Module 02 covers in the next lesson.
