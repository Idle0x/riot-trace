## Why Scope Is the Most Important Concept in This Module

The `ReferenceError: X is not defined` is the most common error in JavaScript. You have seen it. You have fixed it by trial and error — moving code around until the error went away without fully understanding why. After this lesson, you will never fix a scope error by guessing again. You will read the error, locate the scope boundary that caused it, and know exactly what to change.

Scope also explains React's most confusing behavior: why a variable inside a `useEffect` callback sees an old value even though the state has updated. That concept — called a stale closure — is impossible to understand without first understanding scope. This lesson is the prerequisite.

---

## What Scope Means

Scope is the region of code where a variable is accessible. Every variable you declare exists within a scope. Outside that scope, the variable does not exist — attempting to access it produces `ReferenceError`.

JavaScript has three types of scope. You need to understand all three.

---

## Global Scope

Variables declared at the top level of a file, outside of any function or block, are globally scoped. They are accessible from anywhere in that file:

```javascript
const NETWORK = "mainnet";          // global scope
let totalNodeCount = 201;           // global scope

function displayStatus() {
  console.log(NETWORK);             // ✓ accessible — global scope
  console.log(totalNodeCount);      // ✓ accessible — global scope
}

displayStatus();
console.log(NETWORK);               // ✓ accessible here too
```

Global variables sound convenient, but they come with a cost. Every function in every file can read and — if declared with `let` or `var` — modify them. In a codebase with 50 files and 500 functions, a global `let` variable can be modified from anywhere, making it nearly impossible to track down why a value changed. This is why experienced developers minimize global variables and why React's state management system exists — it controls access to shared values.

For this module, you will use global scope only to demonstrate the scope chain. In real application code, use global scope sparingly and make those variables `const` whenever possible.

---

## Function Scope

Variables declared inside a function are **function-scoped** — they exist only for the lifetime of that function call:

```javascript
function processNode(node) {
  const healthScore = node.health * 0.35;  // function-scoped
  const storageScore = node.storage * 0.30; // function-scoped
  const total = healthScore + storageScore;  // function-scoped

  return total;
}

processNode({ health: 95, storage: 250 });
console.log(healthScore);  // ReferenceError: healthScore is not defined
```

The variable `healthScore` was created when `processNode` started executing. When `processNode` returned, JavaScript cleaned it up. It is gone. The `console.log` outside the function tries to access a variable that no longer exists in any scope it can reach.

This is not a limitation. It is a feature. It means each function call is isolated. The `healthScore` in one call to `processNode` does not interfere with the `healthScore` in another call. You can call the same function 1000 times and each call gets its own fresh set of local variables.

---

## Block Scope

Variables declared with `const` or `let` inside any `{ }` block are **block-scoped** — they exist only inside that block:

```javascript
if (true) {
  const blockVariable = "I exist inside this if block";
  let alsoBlock = "me too";
  console.log(blockVariable);  // ✓ accessible
}

console.log(blockVariable);  // ReferenceError: blockVariable is not defined
// The block ended. The variable was cleaned up.
```

This applies to `if` blocks, `for` loops, `while` loops, `try/catch` blocks — any `{ }` creates a new scope for `const` and `let` variables declared inside:

```javascript
for (let i = 0; i < 5; i++) {
  // i exists here
}
console.log(i);  // ReferenceError — i died when the loop ended

for (let index = 0; index < nodes.length; index++) {
  const currentNode = nodes[index];  // new const per iteration
  console.log(currentNode.health);   // ✓ accessible here
}
// currentNode is gone here
```

Note: `var` does **not** respect block scope. This is one of the main reasons `var` is avoided:

```javascript
if (true) {
  var leaked = "var ignores block scope";
  let confined = "let respects block scope";
}
console.log(leaked);   // "var ignores block scope" — it escaped
console.log(confined); // ReferenceError — as expected
```

## The Scope Chain — Looking Upward

When JavaScript looks up a variable, it searches the current scope first. If it does not find it there, it searches the parent scope. Then the grandparent. It keeps going upward until it either finds the variable or reaches the global scope and throws `ReferenceError`.

This chain of scopes is called the **scope chain**:

```javascript
const global = "I'm at the top";

function outer() {
  const middle = "I'm in outer";

  function inner() {
    const local = "I'm in inner";

    // inner() can see all three: its own scope, outer's scope, and global scope
    console.log(local);   // ✓ found in inner's own scope
    console.log(middle);  // ✓ not in inner, found in outer's scope
    console.log(global);  // ✓ not in inner or outer, found at global scope
  }

  inner();

  // outer() cannot see inner's scope — it only looks up, not down
  // console.log(local);  // ReferenceError
}

outer();
```

**The key insight: scope chains only look upward (outward), never downward (inward).**

Inner scopes can see outer scopes. Outer scopes cannot see inner scopes. Think of it as one-way glass: sitting inside a room, you can look out through the glass and see the larger room beyond. But someone standing in the larger room cannot see inside yours.

## Variable Shadowing

What happens when an inner scope declares a variable with the same name as an outer scope?

```javascript
const network = "mainnet";

function checkNetwork() {
  const network = "devnet";  // different variable — same name
  console.log(network);      // "devnet" — the inner one shadows the outer one
}

checkNetwork();
console.log(network);        // "mainnet" — the outer one is unchanged
```

This is called **shadowing**. The inner `network` does not modify the outer `network` — it creates an entirely separate variable that happens to have the same name. Inside `checkNetwork`, any reference to `network` resolves to the inner one. The outer one is invisible from inside the function.

Shadowing is a source of bugs when it happens accidentally. If you intended to modify the outer `network` but accidentally declared a new one with `const` or `let`, the outer variable stays unchanged and you spend 20 minutes wondering why nothing updated.

The professional habit: within a function, if you are intentionally reading or modifying an outer variable, do not redeclare it. If you need a new independent variable, give it a different name.

## Lexical Scope — Where You Write It, Not Where You Call It

This concept is the foundation of closures, which you will encounter in Lesson 22 and in every React hook you use. Understand it now.

A function's scope chain is determined by **where the function is written in the source code**, not by where or how it is called.

```javascript
const value = "outer";

function readValue() {
  console.log(value);  // reads from the scope where readValue was WRITTEN
}

function wrapper() {
  const value = "inner";  // local to wrapper — does not affect readValue's scope chain
  readValue();            // calling readValue from inside wrapper
}

wrapper();  // logs "outer" — not "inner"
```

`readValue` was written at the global scope level. Its scope chain goes: own scope → global scope. It has no awareness of `wrapper`'s scope — even when called from inside `wrapper`. The `value` it sees is always the global `"outer"`.

This is called **lexical scope** (or static scope). "Lexical" means "based on the text of the code." Where you write the function determines what it can see.

This will become deeply important in Tier 3 when you work with `useEffect` and React hooks. A callback function passed to `useEffect` closes over the values from the scope where the component function wrote it. Even if that component re-renders and the values change, the callback sees the values from its original scope — unless you tell React to recreate it. That is the stale closure bug explained. Understanding lexical scope now means the stale closure concept will make immediate sense in Tier 3.

## Practical: Reading Scope Errors

The `ReferenceError: X is not defined` always means one thing: the variable `X` does not exist in any scope that the current line of code can reach. The solution always involves one of these fixes:

**Fix A — The variable is in the wrong scope.** Move the declaration to a higher scope:
```javascript
// Problem: result is declared inside the function
function calculate() {
  const result = 1024 * 8;
}
console.log(result);  // ReferenceError

// Fix A: declare outside, assign inside
let result;
function calculate() {
  result = 1024 * 8;  // modifies the outer result
}
calculate();
console.log(result);  // 8192
```

**Fix B — Return the value instead of expecting the outer scope to see it:**
```javascript
function calculate() {
  const result = 1024 * 8;
  return result;  // send it out to the caller
}
const result = calculate();  // catch it at the outer scope
console.log(result);  // 8192
```

Fix B is usually the cleaner approach. Functions that compute a value and return it are easier to test, easier to reuse, and do not rely on modifying outer-scope variables.

## Common Mistakes

**Mistake 1 — Expecting to see a variable from inside a called function:**
```javascript
function setup() {
  const config = { timeout: 15000 };
}
setup();
console.log(config);  // ReferenceError — config lives in setup's scope
// Fix: either return config, or declare it outside setup
```

**Mistake 2 — Accidentally shadowing instead of modifying:**
```javascript
let count = 0;
function increment() {
  const count = 1;  // shadows the outer count — creates a new one
  // The outer count is still 0. This function did nothing useful.
}
increment();
console.log(count);  // 0 — the outer count was never changed
// Fix: remove the 'const' — just write: count = count + 1;
```

**Mistake 3 — var and block scope:**
```javascript
for (var i = 0; i < 3; i++) {
  // loop body
}
console.log(i);  // 3 — var leaked out of the for loop block
// Use let instead: for (let i = 0; i < 3; i++)
// Then console.log(i) would throw ReferenceError — as expected
```

**Mistake 4 — Nested functions reaching down:**
```javascript
function outer() {
  function inner() {
    const secret = "only inner knows";
  }
  inner();
  console.log(secret);  // ReferenceError — outer cannot see inside inner
}
// Scope chains go UP, not DOWN.
```
