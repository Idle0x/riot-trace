## The Test

Module 02 covered three interconnected concepts: how to declare variables with the right keyword and mutation behavior, how scope determines what a variable can see, and how hoisting changes the order JavaScript appears to read your code.

This boss fight requires all three at once. Not in sequence — in a single cohesive program where each part depends on the previous.

---

## What This Tests

**Requirement 1** tests that you can make the right declaration choices and understand that constants declared at outer scope are readable from inner functions.

**Requirement 2** tests scope understanding at a deeper level. `registerNodes` must read `NETWORK` from the outer scope (not shadow it with a local redeclaration), and it must *modify* `registeredNodes` across multiple calls. If you declare a new `let registeredNodes` inside the function, the outer variable never changes — the second call logs `2` instead of `203`. The Lesson 6 Drill 4 was training you for exactly this.

**Requirement 3** tests hoisting. The `runAudit()` call must appear above the `function runAudit()` definition in your source code. If the call is below the definition, you are not demonstrating hoisting — you are just calling a function in the normal order. The validator checks the output but the mental discipline of placing the call first is the actual skill being tested.

---

## Approach

Do not write all four requirements at once and then try to debug four things simultaneously. Build and verify each requirement separately:

1. Write the three outer-scope declarations and log `VERSION`. Verify Line 1 is correct.
2. Write `registerNodes` and the two calls. Verify Lines 2 and 3 are correct. If Line 3 shows `2` instead of `203`, your function has a scope error.
3. Write the `runAudit()` call (before its definition). Then write the function definition below it. Verify Line 4.

Then run the full sequence and check all four outputs match exactly.
