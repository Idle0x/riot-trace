## Why Hoisting Confuses Everyone

You have written code where a function is defined at the bottom of a file but called at the top, and it worked. You have also written code where a variable seemed to exist before its declaration and returned `undefined` instead of throwing an error. If either of those happened and you did not understand why, this lesson is the explanation.

Hoisting is not magic. It is the result of JavaScript processing your code in two passes before executing it. Understanding those passes makes the behavior completely predictable — and gives you a clear reason to prefer `const` and `let` over `var`.

---

## The Two Passes — How JavaScript Actually Runs Your Code

When you execute a JavaScript file or script, the engine does not start at line 1 and run straight through. It does two passes:

**Pass 1 — The Compilation Pass:** The engine scans the entire file and registers all declarations. It finds every `function`, `var`, `let`, `const`, and `class` declaration. Nothing is executed. Values are not assigned. The engine is just building a map of what exists.

**Pass 2 — The Execution Pass:** The engine runs the code line by line from top to bottom. Now it executes expressions, assigns values, calls functions.

"Hoisting" refers to what happens during Pass 1. Certain declarations are moved — "hoisted" — to the top of their scope. This happens invisibly. You do not see it in your code. But its effects are real.

---

## Function Declarations Are Fully Hoisted

Function declarations — the `function name() { }` syntax — are hoisted completely. Both the name and the entire function body are registered during Pass 1. This means you can call a function before its declaration in the source code:

```javascript
// This works — the call is before the declaration
initialize();

function initialize() {
  console.log("System starting...");
}
```

What JavaScript "sees" after hoisting:
```javascript
// JavaScript registers this during Pass 1:
function initialize() {
  console.log("System starting...");
}

// Then executes this during Pass 2:
initialize();  // works — the function is already registered
```

This behavior is intentional. It allows you to organize files so that the high-level logic appears at the top and the helper function definitions appear at the bottom. Many developers prefer this style because it lets you read the main flow without wading through implementation details.

In your xandeum-pulse codebase, this pattern appears in utility files where `main()` or a primary export is at the top and the helper functions it depends on are defined below. Hoisting makes that readable organization possible.

---

## var Is Hoisted — But Only the Declaration

The `var` keyword is hoisted, but **only the declaration, not the initialization**. This distinction causes the most confusing behavior in JavaScript:

```javascript
console.log(nodeCount);  // undefined — not a ReferenceError, not 201
var nodeCount = 201;
console.log(nodeCount);  // 201
```

What JavaScript actually processes:
```javascript
// Pass 1 hoists this:
var nodeCount;  // declaration — registered, set to undefined

// Pass 2 executes line by line:
console.log(nodeCount);  // undefined — declaration exists, but no value assigned yet
nodeCount = 201;          // initialization — now it has a value
console.log(nodeCount);  // 201
```

The variable `nodeCount` exists in memory from the start of execution (its declaration was hoisted), but its value is `undefined` until the initialization line is reached.

This is dangerous behavior. The code does not crash — it silently gives you `undefined` where you might expect an error or the actual value. Silent failures are worse than loud ones because they lead to bugs that only appear in specific execution paths and are extraordinarily difficult to reproduce.

**The practical consequence:** If you use `var` and accidentally reference a variable before its declaration, JavaScript will not tell you something is wrong. It will return `undefined` and let you continue with broken data.

---

## The Temporal Dead Zone — Why const and let Are Better

`const` and `let` behave differently. They are registered during Pass 1 (their names are noted), but they are **not accessible** until the execution reaches their declaration line. Attempting to access them before that line throws a `ReferenceError`:

```javascript
console.log(health);  // ReferenceError: Cannot access 'health' before initialization
const health = 95;
```

The period between the start of the scope and the declaration line is called the **Temporal Dead Zone (TDZ)**. The variable exists in the engine's memory but is explicitly off-limits.

Compare the two failures:
```javascript
// var — silent, misleading
console.log(nodeCount);  // undefined — gives you a value, but wrong one
var nodeCount = 201;

// const/let — loud, accurate
console.log(health);  // ReferenceError: Cannot access 'health' before initialization
const health = 95;
```

The `ReferenceError` from `const`/`let` tells you exactly what happened: you accessed something before it was ready. The error message even tells you the variable name and the exact problem. You fix it in seconds.

The `undefined` from `var` tells you nothing useful. Your code continues running with the wrong value. The bug shows up three function calls later in a completely different file. You spend an hour debugging.

**Loud errors are professional. Silent failures are dangerous.** This is why the entire JavaScript community moved away from `var` after ES6 introduced `const` and `let` in 2015.

---

## Function Expressions Are Not Hoisted

There is a crucial distinction between **function declarations** and **function expressions**:

```javascript
// FUNCTION DECLARATION — hoisted completely
function greet() {
  console.log("Hello");
}

// FUNCTION EXPRESSION — NOT hoisted (const holds a reference to a function)
const greet = function() {
  console.log("Hello");
};

// ARROW FUNCTION EXPRESSION — also NOT hoisted
const greet = () => {
  console.log("Hello");
};
```

Only function declarations (the `function name() { }` syntax) are fully hoisted. Function expressions stored in variables follow the same rules as the variable keyword used:

```javascript
// var — declaration hoisted, initialization not
console.log(greet);  // undefined — not the function, not an error
var greet = function() { console.log("Hello"); };
greet();             // works — now it is initialized

// const/let — Temporal Dead Zone
greet();             // ReferenceError: Cannot access 'greet' before initialization
const greet = () => { console.log("Hello"); };
```

In modern JavaScript, most functions you write are arrow function expressions stored in `const`. These are not hoisted. Do not call them before their declaration. The error you get if you do will be clear and immediately fixable.

---

## When to Rely on Hoisting — and When Not To

Hoisting is not a bug to avoid. It is a feature with a specific appropriate use:

**Use hoisting intentionally:** When you write function declarations that you want to be callable anywhere in a file, hoisting is working for you. This is why helper functions can go at the bottom of a file while the code that uses them is at the top.

**Do not rely on var hoisting:** There is no good reason to access a `var` variable before its declaration. If you find yourself in a situation where code wants to run before a variable is declared, the fix is to restructure the code — not to rely on `var`'s `undefined` behavior.

**Use const/let consistently:** Their TDZ behavior enforces a good discipline: declare variables before you use them. This makes code more readable and prevents the silent `undefined` bugs that `var` allows.

---

## Class Declarations and Hoisting

A brief note for when you encounter classes in later tiers: class declarations are technically hoisted but behave like `let` — they are in the Temporal Dead Zone until their declaration. You cannot use a class before it is defined:

```javascript
const node = new Node();  // ReferenceError: Cannot access 'Node' before initialization
class Node {
  constructor(health) { this.health = health; }
}
```

## Common Mistakes

**Mistake 1 — Thinking all functions are hoisted:**
```javascript
// Works — function declaration
greet();
function greet() { console.log("hello"); }

// Fails — function expression in const
greet();  // ReferenceError
const greet = () => { console.log("hello"); };
```

**Mistake 2 — Using var and expecting initialization to be hoisted:**
```javascript
console.log(tier);  // undefined — not "Code Literacy"
var tier = "Code Literacy";
// The assignment does not hoist, only the declaration
```

**Mistake 3 — Expecting let/const to behave like function declarations:**
```javascript
console.log(score);  // ReferenceError — not undefined
let score = 0;
// let and const do not silently give undefined — they throw an error
```

**Mistake 4 — Confusing function declarations with function expressions:**
```javascript
// These look similar but hoist differently:
function processNode() { }   // declaration — hoisted completely
const processNode = () => {} // expression — not hoisted
```
