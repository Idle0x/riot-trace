## Why Single Conditions Are Rarely Enough

In your xandeum-pulse health scoring system, a node is not simply "healthy" or "unhealthy" based on one metric. A node is considered critical when its health score is below 50 **and** it is on mainnet **and** its version is outdated. Any one of those conditions alone might be acceptable. All three together triggers an alert.

Real conditions in production code almost never check a single thing. They combine multiple checks. This lesson gives you the three operators for doing that precisely.

---

## The AND Operator — &&

`&&` returns `true` only when **both** sides are truthy. If either side is falsy, the result is `false`.

```javascript
true && true    // true
true && false   // false
false && true   // false
false && false  // false
```

In practice:

```javascript
const health = 47;
const network = "mainnet";

if (health < 50 && network === "mainnet") {
  console.log("Critical mainnet node");
}
// Both must be true for this to log
// health < 50 → true
// network === "mainnet" → true
// true && true → true → the block runs
```

```javascript
const health = 90;
const network = "mainnet";

if (health < 50 && network === "mainnet") {
  console.log("Critical mainnet node");
}
// health < 50 → false
// false && anything → false → the block does not run
// JavaScript does not even evaluate the right side — this is short-circuit
```

---

## The OR Operator — ||

`||` returns `true` when **at least one** side is truthy. Both can be true. Only one needs to be.

```javascript
true || false   // true
false || true   // true
true || true    // true
false || false  // false
```

In practice:

```javascript
const isAdmin = false;
const userXP = 6200;

if (isAdmin || userXP >= 5000) {
  console.log("ACCESS GRANTED");
}
// isAdmin → false
// userXP >= 5000 → true
// false || true → true → access granted
```

---

## The NOT Operator — !

`!` inverts a boolean. `true` becomes `false`. `false` becomes `true`.

```javascript
!true    // false
!false   // true
```

In practice:

```javascript
const isOffline = false;

if (!isOffline) {
  console.log("Node is reachable");
}
// !false → true → the block runs

const isOnline = true;
if (!isOnline) {
  console.log("Node is down");
}
// !true → false → the block does not run
```

`!` is most useful for improving readability. Compare:

```javascript
// Without !:
if (isLoading === false) { }

// With !:
if (!isLoading) { }
// Reads as "if not loading" — more natural English
```

## Short-Circuit Evaluation — How && and || Actually Work

`&&` and `||` do not simply return `true` or `false`. They return one of their **operands** — and they stop evaluating as soon as the result is determined.

### && returns the first falsy value, or the last value if all are truthy:

```javascript
"mainnet" && 201      // 201 — both truthy, returns last
0 && "mainnet"        // 0 — 0 is falsy, stops and returns it
null && "anything"    // null — null is falsy, stops immediately
"yes" && "also yes"   // "also yes" — both truthy, returns last
```

### || returns the first truthy value, or the last value if all are falsy:

```javascript
null || "mainnet"     // "mainnet" — null is falsy, tries right side, returns it
"mainnet" || "devnet" // "mainnet" — first one is truthy, stops immediately
0 || false            // false — both falsy, returns last
```

This behavior enables two patterns you will use constantly:

**Pattern 1 — Default values with ||:**
```javascript
const config = userConfig.port || 3000;
// If userConfig.port is undefined (falsy), use 3000
// If userConfig.port is 8080 (truthy), use 8080

const network = selectedNetwork || "mainnet";
// "If no network selected, default to mainnet"
```

**Pattern 2 — Conditional execution with &&:**
```javascript
// In React (you will write this hundreds of times in Tier 3):
{isLoading && <Spinner />}
// If isLoading is true, render <Spinner />
// If isLoading is false, && short-circuits and nothing renders

// In regular JavaScript:
nodes.length > 0 && console.log("nodes found");
// Only logs if there are nodes
```

## Operator Precedence — Which Evaluates First

When `&&` and `||` appear in the same expression, `&&` evaluates first (higher precedence), then `||`:

```javascript
// What does this mean?
a || b && c

// It means:
a || (b && c)
// NOT: (a || b) && c
```

In practice:

```javascript
const isAdmin = false;
const isOwner = true;
const hasValidToken = true;

if (isAdmin || isOwner && hasValidToken) {
  console.log("authorized");
}
// Evaluates as: isAdmin || (isOwner && hasValidToken)
// isOwner && hasValidToken → true && true → true
// isAdmin || true → false || true → true
// Block runs

// If you meant (isAdmin || isOwner) && hasValidToken:
if ((isAdmin || isOwner) && hasValidToken) { }
// Use parentheses to make your intent explicit
```

**Rule: when mixing && and ||, always use parentheses.** Relying on precedence rules makes conditions hard to read and easy to get wrong. Parentheses are free.

## The Double Negation !! — Converting to Boolean

A useful trick: `!!` converts any value to its explicit boolean equivalent:

```javascript
!!0            // false — 0 is falsy
!!""           // false — empty string is falsy
!!null         // false — null is falsy
!!undefined    // false — undefined is falsy
!!"mainnet"    // true — non-empty string is truthy
!!201          // true — non-zero number is truthy
!!{}           // true — empty object is truthy
```

Used when you want to store whether something exists rather than the value itself:

```javascript
const hasNodes = !!nodes.length;  // true or false, not a number
const isAuthenticated = !!user;   // true or false, not the user object
```

## Nullish Coalescing — ?? (The Smarter Default)

The `||` default pattern has a flaw:

```javascript
const nodeCount = 0;
const displayCount = nodeCount || "unknown";
// displayCount is "unknown" — but 0 is a valid count!
// || treats 0 as falsy and replaces it
```

The `??` operator (nullish coalescing) only triggers for `null` and `undefined` — not for `0`, `""`, or `false`:

```javascript
const nodeCount = 0;
const displayCount = nodeCount ?? "unknown";
// displayCount is 0 — ?? respects the value 0

const health = null;
const displayHealth = health ?? "N/A";
// displayHealth is "N/A" — null triggers ??
```

**When to use `||` vs `??`:**
- `||` — when `0`, `""`, and `false` are also invalid/empty values you want to replace
- `??` — when only `null` and `undefined` should trigger the default (0 and "" are valid)

In most cases involving optional data from an API, `??` is the right choice.

## Common Mistakes

**Mistake 1 — && when || is needed (or vice versa):**
```javascript
// This requires BOTH to be true — probably too strict:
if (isAdmin && hasPermission) { }

// This allows EITHER — might be too permissive:
if (isAdmin || hasPermission) { }

// Read your condition out loud. "If admin AND has permission" vs
// "If admin OR has permission" — which did you mean?
```

**Mistake 2 — Not using parentheses with mixed && and ||:**
```javascript
// Ambiguous and error-prone:
if (a || b && c || d) { }

// Explicit and readable:
if (a || (b && c) || d) { }
```

**Mistake 3 — Using || for defaults when 0 is a valid value:**
```javascript
const timeout = userSetting || 5000;
// If userSetting is 0 (a valid setting!), this returns 5000
// Fix:
const timeout = userSetting ?? 5000;
```

**Mistake 4 — Forgetting that ! converts to boolean:**
```javascript
const nodes = [];
if (!nodes) { console.log("no nodes"); }
// This never logs — [] is truthy, ![] is false
// To check for empty array: if (!nodes.length) { }
```
