## The Test

Module 03 covered how JavaScript makes decisions. `if/else if/else` for multi-path branching based on ranges. `switch` for matching one value against multiple exact cases. Ternary for producing a value from a simple two-path decision. Logical operators (`&&`, `||`, `!`) for combining conditions. All four tools. This boss fight uses all four in one program.

---

## What This Tests

**Requirement 1** tests that your `if/else if` chain is ordered correctly and uses the right thresholds. Health 47 is below 50 — it must reach the final branch without matching any of the higher thresholds.

**Requirement 2** tests that your `switch` has the correct case strings, proper `break` statements, and a functioning `default`. One missing `break` will cause fall-through. The output string format — `"mainnet: PRODUCTION"` — must be exact.

**Requirement 3** tests that you know when to use a ternary. The condition is a `!==` check — if the versions do not match, there is an update available.

**Requirement 4** tests the `!` operator directly. `isUnderMaintenance` is `false`. Negating it with `!` produces `true`. The `if` block runs and logs `"REACHABLE"`.

**Requirement 5** is the integration challenge. You must count how many boolean conditions are true and produce a dynamic count — not a hardcoded number. One approach: convert each condition to a number using `Number(condition)` or a ternary, then add them:

```javascript
const issueCount = Number(condition1) + Number(condition2) + Number(condition3);
// Number(true) === 1, Number(false) === 0
```

Another approach: use a `let` counter and increment it with `if` statements. Either works.

The deliberate misdirection in the requirements — "the answer is 3... wait, no, re-read the values" — is intentional. Read the actual data before assuming. `storage` is `250`. The threshold is `< 100`. `250 < 100` is `false`. That is not an issue. The answer is `2`.

**The em dash in the final line is real.** The character `—` is a Unicode em dash (U+2014), not two hyphens. Copy it from the requirements if needed.

---

## Approach

Build each requirement one at a time, verify the output is correct, then move to the next. Do not attempt to write all five requirements at once.

After you have all five outputs correct individually, verify the complete sequence matches the expected output exactly — five lines, exact strings, correct order.
