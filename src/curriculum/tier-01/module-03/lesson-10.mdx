## When if/else Becomes Noise

Look at this code from your riot-sop transaction categorizer:

```javascript
let displayLabel;
if (transaction.type === "inflow") {
  displayLabel = "IN";
} else {
  displayLabel = "OUT";
}
```

Six lines. One decision. One result stored in a variable. The `if/else` structure here is not adding clarity — it is adding length. The ternary operator and the switch statement are JavaScript's tools for expressing decisions concisely when `if/else` is overkill.

Concise does not mean clever. The goal is removing noise without removing meaning.

---

## The Ternary Operator

The ternary is a compressed `if/else` that produces a value:

```javascript
condition ? valueIfTrue : valueIfFalse
```

Read it as: "if condition, then valueIfTrue, otherwise valueIfFalse."

```javascript
// if/else version (6 lines):
let status;
if (health > 80) {
  status = "ONLINE";
} else {
  status = "DEGRADED";
}

// Ternary version (1 line):
const status = health > 80 ? "ONLINE" : "DEGRADED";
```

The result of a ternary is a value — it can be stored in a variable, returned from a function, passed as an argument, or embedded in a template literal:

```javascript
// Stored:
const badge = xp >= 5000 ? "ELITE" : "STANDARD";

// Returned:
function getLabel(health) {
  return health > 80 ? "healthy" : "at risk";
}

// In a template literal:
console.log(`Node status: ${health > 80 ? "ONLINE" : "DEGRADED"}`);

// As a function argument:
console.log(health > 80 ? "ONLINE" : "DEGRADED");
```

---

## When to Use Ternary

The ternary is appropriate when:

1. You are choosing between two values based on a condition
2. The condition fits comfortably on one line
3. Both the true and false branches are simple values — not complex logic

```javascript
// Good ternary use — simple value assignment:
const color = isOnline ? "green" : "red";
const prefix = isAdmin ? "[ADMIN]" : "[USER]";
const label = version === latest ? "CURRENT" : "OUTDATED";

// Bad ternary use — complex branches that need their own lines:
const result = nodes.length > 0
  ? nodes.filter(n => n.health > 80).map(n => n.id).join(", ")
  : fetchDefaultNodes().then(d => d.slice(0, 5));
// This is unreadable. Use if/else here.
```

The test: can you read the ternary out loud as a sentence without hesitating? If yes, use it. If no, use `if/else`.

---

## Never Nest Ternaries

Nested ternaries — ternaries inside ternaries — are a readability disaster that professional code reviews will reject:

```javascript
// Never do this:
const rank = xp >= 5000 ? "Elite" : xp >= 2500 ? "Senior" : xp >= 1000 ? "Operator" : "Recruit";

// This is what that actually says — nearly impossible to parse:
const rank = xp >= 5000
  ? "Elite"
  : (xp >= 2500
    ? "Senior"
    : (xp >= 1000
      ? "Operator"
      : "Recruit"));
```

For more than two outcomes, use `if/else if/else` or `switch`. That is exactly what those tools are for.

## The Switch Statement

`switch` is designed for one specific situation: checking **one value** against **many possible cases**. It is cleaner than a long `if/else if` chain when you are comparing one variable to multiple exact values.

```javascript
switch (valueToCheck) {
  case possibleValue1:
    // runs if valueToCheck === possibleValue1
    break;
  case possibleValue2:
    // runs if valueToCheck === possibleValue2
    break;
  default:
    // runs if no case matched
}
```

Concrete example:

```javascript
const network = "mainnet";

switch (network) {
  case "mainnet":
    console.log("Production network");
    break;
  case "devnet":
    console.log("Development network");
    break;
  case "testnet":
    console.log("Testing network");
    break;
  default:
    console.log("Unknown network: " + network);
}
// Output: Production network
```

---

## The break Statement — Never Forget It

Without `break`, execution **falls through** to the next case and keeps running until it hits a `break` or the end of the switch:

```javascript
const network = "mainnet";

switch (network) {
  case "mainnet":
    console.log("Production network");
    // no break here!
  case "devnet":
    console.log("Development network");
    break;
  case "testnet":
    console.log("Testing network");
    break;
}
// Output:
// Production network
// Development network   ← fell through from mainnet case!
```

This behavior — fall-through — is almost always a bug. The only intentional use is when multiple cases should share the same code:

```javascript
// Intentional fall-through: mainnet and stagingnet both treated as production
switch (network) {
  case "mainnet":
  case "stagingnet":
    console.log("Production-like environment");
    break;
  case "devnet":
  case "testnet":
    console.log("Development environment");
    break;
}
```

When two empty cases stack like this, it is intentional. When a case has code and no `break`, it is almost always a bug. The rule: every case that has code inside it needs a `break`.

---

## Switch Uses Strict Equality

The comparison in `switch` uses `===` — strict equality. This means types must match:

```javascript
const value = 1;

switch (value) {
  case "1":
    console.log("string one");  // does not match — "1" !== 1
    break;
  case 1:
    console.log("number one");  // matches
    break;
}
// Output: number one
```

If your switch is not matching when you expect it to, check that the case value type matches the switch expression type.

---

## Switch vs if/else if — When to Choose Each

Use `switch` when:
- You are checking one variable against multiple exact values
- All cases are discrete (not ranges)
- You have 3 or more cases

```javascript
// Good switch candidate — discrete string values:
switch (tier) {
  case "Code Literacy": ...
  case "JavaScript Deeply": ...
  case "React Mental Model": ...
}

// Bad switch candidate — ranges (use if/else if instead):
// switch cannot express "greater than 80"
```

Use `if/else if` when:
- You are checking ranges (`health > 80`, `xp >= 5000`)
- Each condition is a different expression, not the same variable against different values
- The conditions involve multiple variables

## Common Mistakes

**Mistake 1 — Missing break (fall-through bug):**
```javascript
switch (status) {
  case "online":
    console.log("online");
    // missing break!
  case "degraded":
    console.log("degraded"); // runs even when status is "online"
    break;
}
```

**Mistake 2 — Missing default:**
```javascript
switch (network) {
  case "mainnet": console.log("production"); break;
  case "devnet": console.log("development"); break;
  // If network is "testnet", nothing happens — silent skip
}
// Always add default to handle unexpected values
```

**Mistake 3 — Ternary with side effects (console.log inside ternary):**
```javascript
// Wrong pattern — ternary produces a value, not an action:
health > 80 ? console.log("healthy") : console.log("at risk");

// Use if/else for side effects:
if (health > 80) { console.log("healthy"); }
else { console.log("at risk"); }

// Use ternary for values:
const label = health > 80 ? "healthy" : "at risk";
console.log(label);
```

**Mistake 4 — Type mismatch in switch:**
```javascript
const code = "1";
switch (code) {
  case 1: console.log("found"); break;  // won't match — "1" !== 1
  case "1": console.log("found"); break; // will match
}
```
